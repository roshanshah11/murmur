# Sparkle 2 integration — Murmur

Implementation notes for wiring [Sparkle 2](https://sparkle-project.org) into Murmur (non-sandboxed AppKit app, distributed as a signed/notarized DMG from GitHub Releases, also installable via the `roshanshah11/homebrew-murmur` cask).

The appcast is hosted at `https://roshanshah11.github.io/murmur/appcast.xml` (GitHub Pages, eventually `https://murmur.app/appcast.xml`).

---

## 1. SwiftPM dependency

Add to `app/Package.swift`:

```swift
// swift-tools-version: 6.0
import PackageDescription

let package = Package(
    name: "Murmur",
    platforms: [
        .macOS(.v13)
    ],
    products: [
        .executable(name: "Murmur", targets: ["Murmur"])
    ],
    dependencies: [
        .package(
            url: "https://github.com/sparkle-project/Sparkle",
            from: "2.6.0"
        ),
    ],
    targets: [
        .executableTarget(
            name: "Murmur",
            dependencies: [
                .product(name: "Sparkle", package: "Sparkle"),
            ],
            path: "Sources/Murmur",
            swiftSettings: [.swiftLanguageMode(.v5)]
        ),
        .testTarget(
            name: "MurmurTests",
            dependencies: ["Murmur"],
            path: "Tests/MurmurTests",
            swiftSettings: [.swiftLanguageMode(.v5)]
        )
    ]
)
```

SwiftPM pulls the binary `Sparkle.framework` (built by the Sparkle team) and links it directly. It will land inside `Murmur.app/Contents/Frameworks/Sparkle.framework/` at build time — verify with `codesign -dvv` post-build.

## 2. Info.plist keys

Murmur is an SPM-built executable, so `Info.plist` is generated by our packaging script (`Scripts/package-app.sh`). Add these keys to whichever plist template that script ingests.

```xml
<key>CFBundleIdentifier</key>
<string>com.murmur.app</string>

<key>SUFeedURL</key>
<string>https://roshanshah11.github.io/murmur/appcast.xml</string>

<key>SUPublicEDKey</key>
<string>REPLACE_WITH_BASE64_PUBLIC_KEY_FROM_GENERATE_KEYS</string>

<key>SUEnableAutomaticChecks</key>
<true/>

<key>SUScheduledCheckInterval</key>
<integer>86400</integer>

<key>NSMicrophoneUsageDescription</key>
<string>Murmur captures audio from your microphone so it can transcribe what you say. Recording is processed entirely on this Mac; nothing is uploaded.</string>

<key>NSAppleEventsUsageDescription</key>
<string>Murmur uses Apple Events to paste the transcribed text into the app you're focused on.</string>
```

### A note on `NSAccessibilityUsageDescription`

The brief mentions `NSAccessibilityUsageDescription`. **There is no such Info.plist key in current macOS** — Accessibility permission is gated by TCC and prompted by AppKit when you call `AXIsProcessTrustedWithOptions(_:)`. You cannot customize that prompt's copy via Info.plist. Including the key is harmless but a no-op; we omit it to keep the plist honest. If product wants custom messaging, do it in our own pre-prompt sheet that the user sees *before* we trigger the system dialog.

## 3. EdDSA key generation

`generate_keys` and `generate_appcast` **do not ship with the SwiftPM Sparkle package** — they only live in the Sparkle binary release tarball. Grab them once:

```sh
# Pin to the version we depend on in Package.swift.
VERSION=2.6.4
curl -fSL "https://github.com/sparkle-project/Sparkle/releases/download/${VERSION}/Sparkle-${VERSION}.tar.xz" \
  | tar -xJ -C /tmp
sudo mv /tmp/bin/generate_keys      /usr/local/bin/sparkle-generate-keys
sudo mv /tmp/bin/generate_appcast   /usr/local/bin/sparkle-generate-appcast
sudo mv /tmp/bin/sign_update        /usr/local/bin/sparkle-sign-update
```

Generate the EdDSA keypair (only once, ever, for the lifetime of the product):

```sh
sparkle-generate-keys
# → Stores private key into the macOS Keychain under "https://sparkle-project.org"
# → Prints the base64 public key to stdout.
```

**Store the private key:**
- Copy the private key from Keychain Access (search for `sparkle`) → export as base64.
- Paste into **1Password → Murmur → Sparkle EdDSA private key** as a secure note. Title field literally that string so the release runbook can find it.
- Also store in the GitHub Actions secret `SPARKLE_ED_PRIVATE_KEY` on the main repo so `release.yml` can sign appcast items in CI.

**Set the public key:** paste the base64 stdout into the `SUPublicEDKey` value in Info.plist (above). Once published in a release, **never rotate this key** without coordinating an emergency forced-update — clients pin to whatever key was in the bundle they installed.

## 4. `SparkleUpdater.swift`

Minimal adapter so the rest of the app never imports Sparkle directly:

```swift
// Sources/Murmur/Updates/SparkleUpdater.swift
import Foundation
import Sparkle

@MainActor
final class SparkleUpdater: NSObject {
    static let shared = SparkleUpdater()

    private let controller: SPUStandardUpdaterController

    override private init() {
        // startingUpdater: true → schedules background checks per SUSchedule…
        // delegate: nil → no custom version comparison, etc.
        // userDriverDelegate: nil → use Sparkle's standard UI.
        self.controller = SPUStandardUpdaterController(
            startingUpdater: true,
            updaterDelegate: nil,
            userDriverDelegate: nil
        )
        super.init()
    }

    /// User-initiated check ("Check for Updates…" menu item).
    func checkForUpdates() {
        controller.checkForUpdates(nil)
    }

    /// Read-only access for menubar wiring that needs to bind `canCheckForUpdates`.
    var updater: SPUUpdater { controller.updater }
}
```

Kick it off at app launch (so the background scheduler starts) — in `AppDelegate.applicationDidFinishLaunching(_:)`:

```swift
_ = SparkleUpdater.shared  // touch to instantiate; startingUpdater:true wires the timer
```

## 5. Menubar "Check for Updates…" item

Sparkle's *Authentic Menu Item* API auto-disables the item while a check is in flight and updates the title — we'll adopt it later. For v1 wire a plain `NSMenuItem`:

```swift
// Inside MenubarController.buildMenu() or wherever the status-bar menu is assembled.
let updateItem = NSMenuItem(
    title: "Check for Updates…",
    action: #selector(MenubarController.checkForUpdates(_:)),
    keyEquivalent: ""
)
updateItem.target = self
menu.addItem(updateItem)
```

```swift
@objc func checkForUpdates(_ sender: Any?) {
    SparkleUpdater.shared.checkForUpdates()
}
```

When ready to upgrade to the auto-title behaviour, replace with:

```swift
import Sparkle
let updateItem = NSMenuItem()
updateItem.bind(NSBindingName("title"),
                to: SparkleUpdater.shared.updater,
                withKeyPath: #keyPath(SPUUpdater.sessionInProgress),
                options: [.valueTransformerName: NSValueTransformerName.negateBooleanTransformerName])
// + the rest of the bindings the Sparkle docs prescribe
```

## 6. Appcast generation & GitHub Pages hosting

`sparkle-generate-appcast` walks a directory of `.dmg` (or `.zip`) bundles, reads each one's `CFBundleShortVersionString` / `CFBundleVersion`, signs each item with the private key from Keychain, and emits an `appcast.xml`.

Pipeline (runs inside `release.yml` after the DMG is notarized & uploaded):

```sh
# 1. Materialise the private key from secrets into the keychain the tool reads.
echo "$SPARKLE_ED_PRIVATE_KEY" | base64 -d > /tmp/sparkle_ed_priv.key
sparkle-sign-update --add-keychain /tmp/sparkle_ed_priv.key   # exact flag varies by version

# 2. Stage every released DMG into one directory.
mkdir -p /tmp/sparkle-feed
cp dist/Murmur-${VERSION}.dmg /tmp/sparkle-feed/
# (Optional) pull older DMGs from previous releases so generate_appcast can
# re-emit historical <item> entries. In practice we just keep a checked-in
# appcast.xml in the docs/ branch and let the tool *append* new items.

# 3. Generate (or update) the appcast.
sparkle-generate-appcast \
  --download-url-prefix "https://github.com/roshanshah11/murmur/releases/download/v${VERSION}/" \
  /tmp/sparkle-feed

# 4. Copy the resulting appcast.xml into the gh-pages publish dir.
cp /tmp/sparkle-feed/appcast.xml docs/appcast.xml
```

How it interacts with Pages:

- `docs/` on `main` (or the `gh-pages` branch — pick one and stick to it) is published via the **Pages → Build from branch** setting.
- Pages serves `appcast.xml` over HTTPS with `Content-Type: application/xml`. Good.
- Each release run regenerates the file. `generate_appcast` is **additive**: it doesn't drop existing `<item>` entries for DMGs that are still in the staged folder. Older entries persist for users on stale versions who do multi-hop upgrades.

**Delta updates are out of scope for v1.** Murmur's DMG is small enough (target ≤ 50 MB) that a full download per release is fine. Revisit when DMG > 200 MB or we start shipping bundled ML models.

## 7. Sandboxing

Sparkle 2's installer launcher uses an XPC service (`Autoupdate.app`, `Updater.app`) that calls privileged install APIs and **cannot run inside the App Sandbox**. That's fine for us: Murmur cannot be sandboxed anyway because it needs:

- a global double-tap-fn hotkey (CGEventTap),
- Accessibility (`AXIsProcessTrusted`) to paste into other apps via CGEvent or AppleScript.

Both require `com.apple.security.app-sandbox = false`. We are non-sandboxed by design, and Sparkle's installer XPC will work out of the box.

**Add to the main repo README:**

> Murmur is a non-sandboxed app. macOS's sandbox cannot host a global hotkey or paste keystrokes into other apps. We compensate by being fully signed with a Developer ID, notarized by Apple, and updated only via Sparkle's EdDSA-signed appcast at `https://roshanshah11.github.io/murmur/appcast.xml`.

## 8. Test-update procedure (v1.0.0 → v1.0.1)

End-to-end smoke test before every release. Run on a clean Mac (or fresh user account).

```sh
# 0. Tag two builds locally.
git checkout main && git tag v1.0.0 && ./Scripts/build-and-package.sh 1.0.0   # → dist/Murmur-1.0.0.dmg
# bump CFBundleShortVersionString to 1.0.1 in your plist template
git tag v1.0.1 && ./Scripts/build-and-package.sh 1.0.1                       # → dist/Murmur-1.0.1.dmg

# 1. Stage v1.0.1 in a temp folder and generate a local appcast.
TMP=$(mktemp -d)
cp dist/Murmur-1.0.1.dmg "$TMP/"
sparkle-generate-appcast \
  --download-url-prefix "file://$TMP/" \
  "$TMP"
ls "$TMP/appcast.xml"

# 2. Point a *debug* SUFeedURL at the local file.
defaults write com.murmur.app SUFeedURL "file://$TMP/appcast.xml"

# 3. Install and launch v1.0.0.
open dist/Murmur-1.0.0.dmg
# → drag Murmur.app to /Applications, launch.

# 4. Menubar → "Check for Updates…".
#    Expected: Sparkle dialog, "Version 1.0.1 is available", Install button.

# 5. Click Install.
#    Expected: progress bar, app quits, relaunches automatically.

# 6. Verify the new version is running.
osascript -e 'tell application "Murmur" to version'    # → "1.0.1"
defaults read com.murmur.app SUFeedURL                  # cleanup-worthy

# 7. Cleanup
defaults delete com.murmur.app SUFeedURL
rm -rf "$TMP"
```

Pass criteria:
- Sparkle dialog appears within ~5 s of clicking the menu item.
- Update applies without user typing an admin password (we install to `/Applications` which the user owns).
- App relaunches automatically.
- `version` reports `1.0.1`.
- No Console errors mentioning `SUSparkleErrorDomain`.

## 9. Common pitfalls

| # | Pitfall | Symptom | Fix |
|---|---|---|---|
| 1 | Appcast served as `text/plain` or `text/html` | "Update information is not signed" or generic parse fail | GitHub Pages serves `.xml` as `application/xml` — fine. If you ever proxy through a CDN, set `Content-Type: application/xml` explicitly. |
| 2 | Missing `sparkle:edSignature` attribute on `<enclosure>` | Sparkle refuses with `SUSignatureError` | `generate_appcast` only signs items if it found the private key in the Keychain. Verify `sparkle-sign-update --verify` against the DMG before publishing. |
| 3 | `SUFeedURL` is `http://` | Sparkle hard-fails (App Transport Security + Sparkle's own check) | Must be `https://`. Pages gives this for free. Custom domain → make sure HTTPS is enabled in Pages settings. |
| 4 | Sparkle not bundled in `Contents/Frameworks/` | `dyld: Library not loaded: @rpath/Sparkle.framework/…` on launch | SwiftPM should embed it automatically because we declare it as a target dependency on an `executableTarget`. If you switch to a Makefile-driven link, copy `Sparkle.framework` into `Murmur.app/Contents/Frameworks/` and re-sign. |
| 5 | Code signature fails because Sparkle's XPC services aren't co-signed | `codesign --verify --deep --strict` reports nested-bundle invalid; notarization rejects | Sign **inner-to-outer**: `XPCServices/*.xpc` → `Updater.app` → `Autoupdate.app` → `Sparkle.framework/Versions/B` → `Sparkle.framework` → `Murmur.app`. Use `--options runtime --timestamp` on every step. Avoid blanket `--deep` (it re-signs nested code with the outer entitlements, which Sparkle's helpers do **not** want). The Sparkle docs ship a `codesign_embedded_executable.sh` reference — adopt it verbatim into `Scripts/codesign.sh`. |
| 6 | `SUPublicEDKey` value has whitespace or trailing newline | Sparkle silently rejects every update | Paste the base64 exactly as `generate_keys` printed it, single line, no surrounding quotes inside the `<string>`. |
| 7 | Same version number in two releases (e.g. shipped `1.0.0` twice) | Clients on `1.0.0` never see the update because Sparkle compares `CFBundleVersion` | Always bump `CFBundleVersion` (build number) **and** `CFBundleShortVersionString` (marketing version). The release workflow should refuse to publish if either is unchanged. |
| 8 | Forgetting to update the Homebrew cask | `brew upgrade --cask murmur` lags Sparkle's in-app updater | Release pipeline runs `tap-bump-script.sh` after the DMG upload — see `.superpowers/drafts/homebrew-tap/tap-bump-script.sh`. |

## User actions owed (one-time)

1. **Enroll in Apple Developer Program** ($99/yr). Generate a *Developer ID Application* cert in Xcode → Settings → Accounts → Manage Certificates. Export `.p12` for CI (`APPLE_DEVELOPER_ID_P12` + `APPLE_DEVELOPER_ID_P12_PASSWORD` secrets).
2. **Generate Sparkle EdDSA keypair** locally with `sparkle-generate-keys` (see §3). Paste public key into the Info.plist template; private key into 1Password and the `SPARKLE_ED_PRIVATE_KEY` GitHub secret.
3. **Create `roshanshah11/homebrew-murmur` repo** (empty, public) before the first release runs.
4. **Enable GitHub Pages** on the main repo: Settings → Pages → Source: `main` / `docs`. Confirm `appcast.xml` is reachable at `https://roshanshah11.github.io/murmur/appcast.xml` after the first publish.
5. **Add notarization credentials** to GitHub Actions secrets: `APPLE_ID`, `APPLE_TEAM_ID`, `APPLE_APP_SPECIFIC_PASSWORD`.
